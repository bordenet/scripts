## Communication Rules

- **No flattery** - Skip "Great question!" or "Excellent point!"
- **No hype** - Avoid "revolutionary", "game-changing", "seamless"
- **Evidence-based** - Cite sources or qualify as opinion
- **Direct** - State facts without embellishment

**Banned phrases**: production-grade, world-class, leverage, utilize, incredibly, extremely, Happy to help!

### рџ”ґ superpowers-plus: GitHub-First Push Policy (NON-NEGOTIABLE)

`superpowers-plus` has two remotes:

| Remote | URL | Role |
|--------|-----|------|
| `upstream` | `https://github.com/bordenet/superpowers-plus.git` | **Source of truth** вЂ” all changes land here first |
| `origin` | `https://gitlab.int.internal.example.net/mbordenet/superpowers-plus.git` | **Private fork** вЂ” pulls from upstream, never receives direct pushes |

**ALL changes go to GitHub (`upstream`) first, via PR в†’ merge into `upstream/main`.** Then pull downstream into the GitLab fork.

- вќЊ **NEVER** push to `origin` (GitLab) first
- вќЊ **NEVER** commit directly to `origin/main`
- вњ… **ALWAYS** branch from `upstream/main` в†’ push to `upstream` в†’ PR on GitHub в†’ merge в†’ pull into `origin/main`

```bash
git fetch upstream
git checkout -b fix/my-change upstream/main
# ... make changes, commit ...
git push upstream fix/my-change
# Create PR on GitHub, merge it, then:
git checkout main && git merge upstream/main && git push origin main
```
